Java Standards

The following are programming standards set by Oracle and other governing bodies concerning the best usage of the java programming language.

OOP Rules

  • A static field or method should be directly referred to by its class name instead of its corresponding object name.
  • An overridden method from an interface or abstract class must be marked with @Override annotation.
  • varargs is recommended only if all parameters are of the same type and semantics. Parameters with Object type should be avoided.
  • Modifying the method signature is forbidden to avoid affecting the caller. A @Depreciated annotation with an explicit description of the new service is necessary when as interface is depreciated.
  • Using a depreciated interface is probated.
  • Since NullPointerException can possibly be thrown while calling the equals method of Object, equals should be invoked by a constant or an object that is definitely not null.
  • Use the equals method, rather than reference equality '==', to compare primitive wrapper classes.
  • Rules for using primitive data types and wrapper classes:
    1. Members of a POJO class must be wrapper classes
    2. The return value and arguments of a RPC method must be wrapper classes
  • While defining POJO classes like DO, DTO, VO, etc., do not assign any default values to the member
  • To avoid a deserialization failure, do not change the serialVersionUID when a serialized class needs to be updated, such as adding some new members. If a completely incompatible update is needed,
    change the value of serialVersionUID in case of a confusion when deserialized.
  • Business logic in constructor methods is prohibited. All in initializations should be implemented in the init method
  • The toString method must be implemented in a POJO class. The super.toString method should be called in the beginning of the implementation if the current class extends another POJO class
  • For a setter method, the argument name should be the same as the field name. Implementations of business logic's in getter/setter methods, which will increase difficulties of the
    troubleshooting, are not recommended.
  • Use the append method in StringBuilder inside a loop body when concatenating multiple strings
  • Keyword final should be used in the following situations:
    1. A class which is not allowed to be inherited, or local variables not ne reassigned
    2. An argument which is not allowed to be modified
    3. A method which is not allowed to be overridden
  • Be cautious to copy an object using the clone method in Object. The default implementation of clone in Object is a shallow copy, which copies fields as pointers to the same objects
    in memory
  • Define the access level of members in class with severe restrictions:
    1. Constructor methods must be private if an allocation using the keyword outside of the class is forbidden
    2. Constructor methods are not allowed to be public or default in a utility class
    3. Nonstatic class variables that are accessed from inheritance must be protected
    4. Nonstatic class variables that no one can access except the class that contains them must be private
    5. Static variables that no one can access except the class that contains them must be private
    6. Static variables should be considered in determining whether they are final
    7. Class methods that no one can access except the class that contains them must be private
    8. Class methods that are accessed from inhabitants must be protected

Java Threads

  • Thead-safe should be ensured when initializing singleton instance, as well as all methods in it
  • A meaningful thread name is helpful to trace the error information, so assign a name when creating threads or thread pools
  • Threads should be provided by thread pools. Explicitly creating threads is not allowed. Using thread pools can reduce the time of creating and destroying thread and save system resource. If we do not use thread pools, lots of similar threads will be created which lead to "running out of memory" or over-switching problem
  • A thread pool should be created by ThreadPoolExecutor rather than Executors. These would make the parameters of the thread pool understandable. It would also reduce the risk of running out of system resources
  • SimpleDateFormat is unsafe, do not define it as a static variable. If you have to, lock or DateUtils class must be used
  • remove() method must be implemented by ThreadLocal variables, especially when using thread pools in which threads are often reused. Otherwise, it may affect subsequent business logic and cause unexpected problems such as memory leaks.
  • In highly concurrent scenarios, performance of Lock should be considered in synchronise calls. A block lock is better than a class lock
  • When adding locks to multiple resources, tables in the database and objects at the same time, locking sequence should be kept consistent to avoid deadlock
  • A lock needs to be used to avoid update failure when modifying one record concurrently. Add lock either in the application layer, in the cache, or add optimistic lock in the database by using version as an update stamp
  • Run multiple TimeTask by using ScheduledExecutorService rather than Timer because Timer because Timer will kill all running threads in case of failing to catch exceptions
  • In concurrent scenarios, one easy solution to optimize the lazy initialization problem by using double-checked locking (referred to The Double-checking locking is broken Declaration), is to declare the object type as volatile

Naming Convention


Package names

  • Package names should be all lower case without underscores or other special characters
  • Don't use plural form. Follow the convention of the standard API which uses for instances
  • java.lang.annotation and not java.lang.annotations
  • Examples: com.yourcompany.widget.button, com.yourcompany.core.api

Class, Interface and Enum Names

  • Class and enum names should typically be nouns
  • Interface names should typically be nouns or adjectives ending with...able
  • Use mixed case with the first letter in each word in upper case (i.e. CamelCase)
  • Match the regular expression ^[A-Z][a-zA-Z0-9]*$
  • Use whole words and avoid using abbreviations unless the abbreviation is more widely used than the long form
  • Format an abbreviation as a word if the word is part of a longer class name
  • Examples: ArrayList, BigInteger, ArrayIndexOutOfBoundsException, Iterable

Method Names

  • Method names should typically be verbs or other descriptions of actions
  • They should match the regular expression ^[A-Z][a-zA-Z0-9]*$
  • Use mixed case with the first letter in lower case
  • Examples: toString, hashCode

Variables

  • Variable names should be in mixed case with the first letter in lower case
  • Match the regular expressions ^[A-Z][a-zA-Z0-9]*$
  • Further recommendation: Variables
  • Examples: elements, currentIndex

Type Variables

  • For simple cases where there are few type variables involved use a single upper case letter
  • Match the regular expressions ^[A-Z][a-zA-Z0-9]*$
  • If one letter is more descriptive than another (such as K and V for keys and values in maps or R for a function return type) use that, otherwise use T
  • For complex cases where single letter type variables become confusing, use longer names written in all capitol letters and use underscore (_) to separate words
  • Examples: T,V,SRC_VERTEX
